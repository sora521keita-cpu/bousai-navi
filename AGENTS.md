# AGENTS.md — 自治体別パーソナル防災ナビ 開発進行ルール

このリポジトリでは、「自治体別パーソナル防災ナビ」MVPを段階的に開発する。

このアプリは、災害通知アプリではない。  
主目的は、自治体別の防災情報、公的資料への導線、緊急時の行動カード、オフライン確認、家族・職場共有を支援することである。

---

1. `docs/00_README_Codex実行概要.md`
2. `docs/06_受け入れテストと品質基準.md`
3. 現在実装する段階の指示ファイル

## 2. 実装前の必須行動

各段階の実装前に、必ず以下を行うこと。

1. 現在のリポジトリ構成を確認する
2. 既存コードがある場合は、破壊的変更を避ける
3. その段階で実装する範囲を5〜10項目で計画する
4. その段階では実装しないことも明示する
5. 不明点がある場合は、実装前に「置いた仮定」を短く列挙する
6. MVPとして単純な実装を選べる場合は、その仮定に基づいて作業を進める
7. 既存構成を大きく破壊する判断が必要な場合は、実装前に確認事項として報告する

現在の段階に含まれない機能は、必要な型・プレースホルダー程度に留め、本実装しないこと。

---

## 3. 基本方針

このアプリの主目的は以下である。

- 自宅・職場・実家などの自治体防災情報を確認しやすくする
- 公的資料・一次情報への導線を整理する
- 緊急時に迷わず行動できる状況別行動カードを提示する
- 通信不能時でも最低限の緊急手順を読めるようにする
- 家族・職場に共有しやすい防災メモを生成する

災害通知、緊急地震速報、災害予測は自前実装しない。

---

## 4. 技術方針

既存構成がない場合は、以下を前提にする。

- React
- TypeScript
- Vite
- モバイルファーストUI
- localStorage
- Service Worker
- 静的データはTypeScriptまたはJSONで管理
- 外部APIなしで動作するMVP

既存リポジトリに別の構成がある場合は、無理に作り替えず、既存構成に合わせる。

## 5. 段階実行ルール

### 第一段階: UI骨格

実装するもの:

- 下部3タブ
  - 地域マップ
  - 防災ライブラリ
  - 緊急
- 緊急タブへ常時アクセスできる固定ボタン
- 防災ライブラリの仮カード
- 緊急行動カードの仮表示
- モバイルファーストの基本レイアウト

この段階では、自治体データ・本格地図・Service Worker・共有機能は実装しない。  
必要な場合でも、後続段階で拡張しやすいプレースホルダーや型の下準備に留める。

### 第二段階: 自治体データと地図

実装するもの:

- 東京都・埼玉県・千葉県・神奈川県のサンプル自治体
- 都道府県選択
- 自治体一覧
- 自治体詳細
- 自宅・職場・実家・よく行く場所として登録
- localStorage保存
- 仮リンクと実リンクの区別

この段階では、全国自治体の完全網羅や現在地からの厳密な自治体判定は実装しない。  
必要な場合でも、将来実装用のボタン・説明・プレースホルダー程度に留める。

### 第三段階: 防災ライブラリ・緊急モード・オフライン

実装するもの:

- 防災ライブラリのカテゴリ
- 信頼度A/B/C/Dラベル
- ユーザー属性フィルター
- 状況別緊急行動カード
- 次の行動への導線
- Service Workerによる最低限のオフライン対応
- オフライン時の注意表示

必ず以下2件を防災ライブラリの初期データに含める。

1. 東京備蓄ナビ「自分に合った備蓄を調べてみよう」
   - URL: https://www.bichiku.metro.tokyo.lg.jp/tool/
   - 信頼度: A
   - 情報源: 東京都

2. 東京都防災ホームページ「東京くらし防災」・「東京防災」の閲覧
   - URL: https://www.bousai.metro.tokyo.lg.jp/1028036/1028051/index.html
   - 信頼度: A
   - 情報源: 東京都

この段階では、リアルタイム災害通知や外部API連携は実装しない。  
オフライン対応は、アプリ本体と緊急行動カードを最低限確認できる範囲に留める。

### 第四段階: 共有と品質改善

実装するもの:

- 家族向け共有テキスト生成
- 職場向け共有テキスト生成
- Web Share API対応
- クリップボードコピー
- 共有APIが使えない場合の手動コピー表示
- アクセシビリティ改善
- localStorage失敗時の最低限のエラーハンドリング
- オフライン時の表示改善

この段階では、ユーザーアカウント、サーバー同期、家族の電話番号管理、個人情報の保存機能は実装しない。

---

## 6. 禁止事項

以下は禁止する。

- 災害予測を行う
- 緊急地震速報を自前実装する
- 現在地情報を無断取得する
- 個人情報をサーバー送信する
- 架空URLを実在リンクのように表示する
- 仮リンクを仮リンクと明示せずに表示する
- 「この避難所に行けば安全です」と断定する
- 「この地域は安全です」と断定する
- 公的情報を改変して断定的に表示する
- 一度に全国自治体完全対応を目指す
- その段階に含まれない機能を先回りして大きく実装する

## 7. 完了条件

各段階の完了時には、必ず以下を確認する。

1. 指示書の受け入れ条件を満たしている
2. TypeScriptエラーがない
3. ビルドが通る
4. 重大なコンソールエラーがない
5. モバイル幅でUIが大きく崩れない
6. 緊急タブへすぐアクセスできる
7. その段階で実装しない機能に踏み込みすぎていない

可能であれば以下を実行する。

```bash
npm run build
```

Lintやテストが用意されている場合は、以下も実行する。

```bash
npm run lint
npm test
```

## 9. 判断に迷った場合の優先順位

判断に迷った場合は、以下の優先順位に従う。

1. 緊急時に迷わないこと
2. 防災情報を断定しすぎないこと
3. 公的情報・一次情報を優先すること
4. MVPとして単純に実装すること
5. 既存コードを壊さないこと
6. 後から差し替えやすいデータ構造にすること
7. UIを複雑にしすぎないこと
